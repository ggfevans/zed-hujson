## 1. Overview

### 1.1 Purpose

Build a Zed editor extension that provides native support for **HuJSON** (Human JSON), a superset of JSON defined by the **JWCC** (JSON With Commas and Comments) specification. HuJSON adds exactly two features to standard JSON:

1. **C-style comments** ‚Äî line comments (`//`) and block comments (`/* */`)
2. **Trailing commas** ‚Äî optional trailing comma after the last element in arrays and objects

All valid JSON is valid HuJSON. HuJSON intentionally rejects all other extensions (unquoted keys, hex literals, `Infinity`/`NaN`, multi-line strings, etc.).

### 1.2 Scope ‚Äî v0.1

| In scope | Out of scope (future) |
|---|---|
| Syntax highlighting via Tree-sitter | Formatter integration (`hujsonfmt`) ‚Äî v0.2 |
| File type association (`.hujson`) | LSP integration ‚Äî v0.3+ |
| Bracket matching | `.jwcc` file association (reconsider if requested) |
| Auto-indentation | Semantic analysis / validation |
| Code outline (object key navigation) | Tailscale ACL-specific intelligence |

- Grammar lifecycle (parser regen, upstream syncs) is decoupled from extension lifecycle (Zed API bumps, language config tweaks).

## 3. Tree-sitter Grammar

### 3.1 Approach

**Fork `tree-sitter/tree-sitter-json`** and make a single targeted modification: allow trailing commas in arrays and objects. The upstream grammar already supports `//` and `/* */` comments.

**Fork from HEAD of `master`** (not an older tag). Comment support was added relatively recently ‚Äî older releases may lack it. Pin the fork to a specific commit SHA in your first commit message for traceability (e.g., "Forked from tree-sitter-json at abc1234"). At time of writing the pinned SHA is `001c28d7a29832b06b0e831ec77845553c89b56d`.

### 3.2 Grammar Configuration Files

After forking, update the following files at the **repo root** (decision 1 ‚Äî the grammar is not nested under `grammars/hujson/`):

**`tree-sitter.json`** (required for `tree-sitter test` and `tree-sitter parse` to work):
```json
{
  "grammars": [
    {
      "name": "hujson",
      "scope": "source.hujson",
      "file-types": ["hujson"],
      "injection-regex": "hujson",
      "highlights": "queries/highlights.scm"
    }
  ]
}
```

**`package.json`** ‚Äî inherited from the fork. Update `name` to `tree-sitter-hujson` and `description` accordingly. The rest (dependencies, build scripts) can remain as upstream.

### 3.3 Grammar Change

The modification is to the `commaSep` / `commaSep1` helper functions in `grammar.js` (at the repo root). The exact implementation will depend on the current shape of the upstream helpers at fork time, but the semantic change is:

**Before (standard JSON):**
```javascript
// Only allows: elem, elem, elem
commaSep(rule) {
  return optional(commaSep1(rule));
},
commaSep1(rule) {
  return seq(rule, repeat(seq(',', rule)));
}
```

**After (JWCC ‚Äî trailing comma tolerated):**
```javascript
// Allows: elem, elem, elem,  (note trailing comma)
commaSep(rule) {
  return optional(seq(rule, repeat(seq(',', rule)), optional(',')));
}
```

### 3.4 Strictness Policy

**Strict HuJSON only.** The grammar MUST NOT accept constructs beyond the JWCC spec:

| Construct | Accept? |
|---|---|
| `//` line comments | ‚úÖ Yes |
| `/* */` block comments | ‚úÖ Yes |
| Trailing commas in `[]` and `{}` | ‚úÖ Yes |
| Standard JSON values (string, number, bool, null) | ‚úÖ Yes |
| Unquoted object keys | ‚ùå No ‚Äî parse error |
| Single-quoted strings | ‚ùå No ‚Äî parse error |
| Hex number literals (`0xFF`) | ‚ùå No ‚Äî parse error |
| `Infinity`, `-Infinity`, `NaN` | ‚ùå No ‚Äî parse error |
| Multi-line strings | ‚ùå No ‚Äî parse error |
| ES5 identifier property names | ‚ùå No ‚Äî parse error |

Since the upstream `tree-sitter-json` already rejects all of these, and our only change is adding trailing comma support, this strictness is inherited automatically ‚Äî no additional work required.

### 3.5 Error Recovery

Tree-sitter provides built-in error recovery via its GLR parsing algorithm. Invalid tokens are wrapped in `ERROR` nodes in the syntax tree, which Zed renders with error highlighting. No custom error recovery logic is needed for v0.1.

**Expected behaviour on invalid input:**

| Input | Expected result |
|---|---|
| `{key: "value"}` (unquoted key) | `key` wrapped in ERROR node, rest of tree parses |
| `{"a": 0xFF}` (hex literal) | `0xFF` wrapped in ERROR, object structure intact |
| `{"a": 1,, "b": 2}` (double comma) | Second `,` wrapped in ERROR |
| Unterminated `/* comment` | ERROR from `/*` to end of file |
| Nested `/* /* */ */` | First `*/` terminates the comment (C-style, no nesting). The trailing ` */` becomes dangling content ‚Üí ERROR node |

## 5. Syntax Queries

> **Path note (post-`cf8d0d1`):** the query files described below now live in the external grammar repo at [`ggfevans/tree-sitter-hujson`](https://github.com/ggfevans/tree-sitter-hujson) under `queries/*.scm` ‚Äî **not** under this repo's `queries/hujson/` (which is removed; see ¬ß14.5). The S-expression contents below remain the source of truth for what the queries should contain. Subsection headings keep the historic `queries/hujson/...` paths for traceability.

All query files use Tree-sitter's S-expression format. These should be adapted from Zed's built-in JSONC queries, adjusted for the HuJSON node names produced by the forked grammar.

### 5.1 Highlights (`queries/hujson/highlights.scm`)

```scheme
; Strings
(string) @string
(escape_sequence) @string.escape

; Numbers
(number) @number

; Booleans and null
(true) @constant.builtin
(false) @constant.builtin
(null) @constant.builtin

; Object keys (string node in pair key position)
(pair
  key: (string) @property.json_key)

; Punctuation
"{" @punctuation.bracket
"}" @punctuation.bracket
"[" @punctuation.bracket
"]" @punctuation.bracket
":" @punctuation.delimiter
"," @punctuation.delimiter

; Comments
(comment) @comment
```

> **Note:** The `@property.json_key` capture name is confirmed correct ‚Äî Zed's built-in JSON and JSONC highlights (`crates/grammars/src/json/highlights.scm`) use this exact name. Node names (`string`, `pair`, `comment`, `escape_sequence`, etc.) should be verified against the actual grammar output after forking and regenerating with `tree-sitter parse` on a sample file.

### 5.2 Brackets (`queries/hujson/brackets.scm`)

```scheme
("{" @open "}" @close)
("[" @open "]" @close)
("\"" @open "\"" @close)
```

### 5.3 Indents (`queries/hujson/indents.scm`)

```scheme
(object) @indent
(array) @indent

"}" @outdent
"]" @outdent
```

### 5.4 Outline (`queries/hujson/outline.scm`)

```scheme
(pair
  key: (string) @name) @item
```

This provides a navigable outline of top-level and nested object keys in Zed's outline panel.

## 6. Data Handling

### 6.1 Input

The extension processes text buffers in the Zed editor. The Tree-sitter parser operates on the raw UTF-8 byte stream. No file I/O is performed by the extension itself ‚Äî Zed handles file reading/writing.

### 6.2 Character Encoding

Tree-sitter and Zed both operate on UTF-8. HuJSON (like JSON per RFC 8259) requires UTF-8 encoding. No special encoding handling is needed.

**Limitation:** If a user opens a UTF-16 or UTF-32 encoded `.hujson` file (rare but possible with legacy tooling), Tree-sitter will silently misparse it ‚Äî producing garbled output with no useful error. This is a Zed-wide limitation, not specific to this extension. No mitigation needed in v0.1, but worth noting for awareness.

### 6.3 Large Files

Tree-sitter's incremental parsing handles large files efficiently ‚Äî only changed regions are re-parsed. No file size limits need to be imposed by the extension. Zed's own large-file handling applies.

## 7. Error Handling Strategy

### 7.1 Grammar-Level Errors

Handled entirely by Tree-sitter's error recovery (see ¬ß3.5). Invalid tokens produce `ERROR` nodes that Zed highlights in red. No custom logic required.

### 7.2 Extension-Level Errors

The v0.1 extension has no runtime logic beyond registration ‚Äî there are no extension-level errors to handle. When formatter/LSP support is added in v0.2+, error handling for process spawning, timeouts, and malformed output will be needed.

### 7.3 Grammar Namespace Collision

The extension registers a grammar named `hujson`, distinct from Zed's built-in `json` and `jsonc` grammars. Since the file extension (`.hujson`) is unique, there should be no conflict ‚Äî Zed dispatches grammars by file extension, not by grammar family.

**Decision 5 (resolved): defence-in-depth verification.** Confirm via three layers, in order:

1. **Manual smoke in Zed** ‚Äî open `.json`, `.jsonc`, and `.hujson` files; confirm each binds to the correct grammar via the status bar; confirm Zed's log shows no duplicate-grammar warnings.
2. **README "Compatibility" section** ‚Äî explicitly state that this extension only handles `.hujson`, that `.json`/`.jsonc` are left to Zed's built-ins, and that the grammar id `hujson` is distinct from `json`/`jsonc`. This sets expectations for users and registry reviewers.
3. **Registry pre-publish search** ‚Äî before opening the PR to `zed-industries/extensions`, grep the registry for `hujson` (id collision), `\.hujson\b`, and `path_suffixes.*hujson` (extension claim). If any prior claim exists, decide on rename / coordination before submitting.

### 7.4 Grammar Generation Errors

If `tree-sitter generate` fails after modifying `grammar.js`, the grammar change has introduced a conflict. Debug with:

```bash
tree-sitter generate --log
tree-sitter parse test-file.hujson --debug
```

CI should catch these before merge (see ¬ß9).

## 8. Testing Plan

### 8.1 Grammar Tests (Tree-sitter Corpus)

Tree-sitter uses a corpus-based testing format. Each test file in `test/corpus/` (at the repo root, per decision 1 ‚Äî not under `grammars/hujson/test/corpus/`) contains named test cases with input and expected syntax tree output.

**Decision 4 (resolved): node-name verification belt-and-braces.** After every `tree-sitter generate`:

1. Audit `src/node-types.json` against every node referenced in `queries/hujson/*.scm` (e.g. `string`, `pair`, `comment`, `escape_sequence`, `number`, `true`, `false`, `null`, `object`, `array`).
2. Run `tree-sitter query queries/hujson/<file>.scm` for each query file; any unknown node name fails the build.

Both checks run locally (via `scripts/check-queries.sh`) and in CI (¬ß9.1).

#### Required Test Cases

**`comments.txt`** ‚Äî Comment handling:
- Line comment at end of line
- Line comment on its own line
- Block comment inline
- Block comment spanning multiple lines
- Comment inside an object (between pairs)
- Comment inside an array (between elements)
- Comment before/after root value
- Nested `/*` inside block comment (C-style, no nesting ‚Äî first `*/` terminates, remainder is ERROR)
- Empty block comment `/* */`

**`trailing_commas.txt`** ‚Äî The core HuJSON addition:
- Trailing comma in object `{"a": 1,}`
- Trailing comma in array `[1, 2, 3,]`
- Trailing comma in nested object `{"a": {"b": 1,},}`
- Trailing comma in nested array `[[1,], [2,],]`
- Trailing comma with comment after `{"a": 1, // note}`
- Comment before trailing comma (comma on next line after comment ‚Äî tests `extras`/whitespace interaction):
  ```json
  {
    "a": 1 // comment
    ,
  }
  ```
- Trailing comma in empty-ish structures ‚Äî `{"a": 1,}` (single element + trailing)
- Multiple trailing commas `{"a": 1,,}` ‚Üí ERROR (only one trailing comma allowed)
- Leading comma `{,"a": 1}` ‚Üí ERROR
- Trailing comma in empty object `{,}` ‚Üí ERROR (no element to trail after)
- Trailing comma in empty array `[,]` ‚Üí ERROR (no element to trail after)

**`literals.txt`** ‚Äî Standard JSON value parsing (inherited from upstream, verify not broken):
- Strings with escape sequences (`\"`, `\\`, `\n`, `\uXXXX`)
- Numbers (integer, float, negative, exponent)
- `true`, `false`, `null`

**`objects.txt`** ‚Äî Object structures:
- Empty object `{}`
- Single pair
- Multiple pairs
- Nested objects
- Object with all value types

**`arrays.txt`** ‚Äî Array structures:
- Empty array `[]`
- Single element
- Multiple elements
- Nested arrays
- Mixed value types

**`degenerate.txt`** ‚Äî Edge cases and boundary conditions:
- Empty file (0 bytes) ‚Äî should parse without crash, produce empty tree
- Whitespace-only file ‚Äî should parse cleanly, no ERROR nodes
- Comment-only file (`// just a comment`, no JSON value) ‚Äî valid HuJSON per spec? Verify against `hujsonfmt` behaviour
- BOM (Byte Order Mark `\uFEFF`) at start of file ‚Äî Tree-sitter should handle or ERROR gracefully
- Bare root value: `"hello"` (string, no object/array wrapper) ‚Äî valid JSON, valid HuJSON
- Bare root value: `42` (number, no wrapper) ‚Äî valid JSON, valid HuJSON
- Bare root value: `null` ‚Äî valid JSON, valid HuJSON
- Unicode in comments: `// Êó•Êú¨Ë™û„Ç≥„É°„É≥„Éà üéâ`
- Unicode in string values: `{"emoji": "üéØ", "cjk": "Êº¢Â≠ó"}`
- Deeply nested structure (50+ levels) ‚Äî verify parser doesn't stack overflow
- Very long single line (>100K characters) ‚Äî verify incremental parsing handles it

**`invalid.txt`** ‚Äî Strictness enforcement (all should produce ERROR nodes):
- Unquoted key: `{key: "value"}`
- Single-quoted string: `{'key': 'value'}`
- Hex literal: `{"a": 0xFF}`
- Infinity: `{"a": Infinity}`
- NaN: `{"a": NaN}`
- Unquoted string value: `{"a": hello}`

Manual testing in Zed with real-world files:

## 9. CI/CD

### 9.1 GitHub Actions ‚Äî CI (`ci.yml`)

Triggered on: push to `main`, pull requests.

**Jobs:**

1. **grammar-test**
   - `npm install -g tree-sitter-cli` (use npm, not cargo ‚Äî faster in CI and consistent with the grammar's Node-based toolchain)
   - `tree-sitter generate` (run from repo root)

### 11.3 Upstream Sync Strategy

The grammar is a fork of `tree-sitter/tree-sitter-json`. Upstream changes (bug fixes, performance improvements, Tree-sitter API updates) need to be merged periodically.

**Process:**

Since the grammar lives in a subfolder of a monorepo, you cannot directly `git merge` the upstream repo ‚Äî the directory structures don't align. Instead:

1. Clone upstream separately for comparison: `git clone https://github.com/tree-sitter/tree-sitter-json.git /tmp/tree-sitter-json`
2. Diff the upstream changes against your fork: `diff -ru /tmp/tree-sitter-json/grammar.js grammars/hujson/grammar.js`
3. Manually port relevant changes into `grammars/hujson/`, preserving the trailing comma modification
4. Re-run `tree-sitter generate` and the full test corpus after any port
5. Commit with a message referencing the upstream commit: "Sync grammar with tree-sitter-json at abc1234"

**Alternative:** If sync frequency increases, consider using `git subtree` to manage the grammar subfolder, which allows direct merging: `git subtree pull --prefix=grammars/hujson upstream-json master --squash`. This requires initial setup with `git subtree add` but simplifies future syncs.

**What could break on upstream sync:**
- **Node name changes** ‚Äî would break all `.scm` query files. Grep queries for renamed nodes.
- **`extras` array changes** ‚Äî could affect comment handling. Verify comments still parse.
- **`commaSep` refactors** ‚Äî our core change. If upstream restructures these helpers, manually reapply the trailing comma logic.
- **`tree-sitter.json` changes** ‚Äî this config file controls scope, file-types, and highlights path. Changes could affect parser generation or Neovim/Helix consumers.
- **Tree-sitter CLI version bumps** ‚Äî may require regenerating `src/parser.c` with a newer CLI.

**Cadence:** Check upstream quarterly, or when a Tree-sitter major version is released. Set a GitHub reminder or Dependabot-style alert.

## 14. Highlighting Fix Addendum (2026-05-06)

> **Status:** Spec drafted, implementation pending. Cross-repo: changes required in both [`ggfevans/zed-hujson`](https://github.com/ggfevans/zed-hujson) and [`ggfevans/tree-sitter-hujson`](https://github.com/ggfevans/tree-sitter-hujson).

### 14.1 Problem Statement

When the extension is installed via Zed's **Install Dev Extension** action against this repository:

| Observed | Expected |
|---|---|
| Status bar shows `HuJSON` for `.hujson` files. | Status bar shows `HuJSON`. ‚úÖ |
| Buffer text is uncoloured (plain). No string colour, no comment colour, no key colour. | Same colouring as JSON, **plus** highlighting for `//` and `/* */` comments. ‚ùå |

The status-bar reading confirms language registration (`extension.toml` + `languages/hujson/config.toml`) is working. The absence of any colouring indicates the grammar is not producing usable parse trees in Zed, OR the Tree-sitter queries the grammar ships are not matching against the parser's actual node names.

This regression was introduced (or exposed) by commit `cf8d0d1` ("restructure: extract grammar to standalone tree-sitter-hujson repo"), which moved the grammar to an external repo without verifying the new fetch-and-load path end-to-end in dev-mode.

### 14.2 Root-Cause Hypotheses

Per the project's debugging protocol ‚Äî list, gather evidence, then fix. **Do not apply speculative fixes.**

**H1 ‚Äî External grammar at pinned commit `69e5e14` is not buildable by Zed.**
- Possible failure modes: `src/parser.c` not committed at that SHA; `parser.c` out of sync with `grammar.js`; `tree-sitter.json` `highlights` path resolves to a missing file; Zed's grammar fetcher silently errors on first load.
- Evidence to gather:
  - `git clone https://github.com/ggfevans/tree-sitter-hujson /tmp/ts-hujson && git -C /tmp/ts-hujson checkout 69e5e147274d91608f146c7ecc6d0821a390d951`
  - Confirm presence of `src/parser.c`, `src/node-types.json`, `src/grammar.json`, `src/tree_sitter/`, `queries/highlights.scm`, `queries/brackets.scm`, `queries/indents.scm`, `queries/outline.scm`, `tree-sitter.json`.
  - `tree-sitter parse /Users/gvns/code/projects/zed-hujson/examples/sample.hujson` (run from `/tmp/ts-hujson`) ‚Äî should produce a tree with no `ERROR` nodes.
  - In Zed: open Dev ‚Üí Open Log; reload the extension; capture grammar-fetch / parser-build / WASM messages.
- Confirms H1 if: log shows fetch/build failure, OR `parser.c` is missing/stale, OR `tree-sitter parse` errors at the pinned SHA.
- Rules out H1 if: clean parse, log silent on errors, all expected files present.

**H2 ‚Äî Queries reference node names absent from the parser's `node-types.json`.**
- Even if the parser builds, mismatched node names mean queries match nothing ‚Üí no captures ‚Üí no highlighting.
- Evidence to gather:
  - At pinned SHA, run `tree-sitter query queries/highlights.scm examples/sample.hujson` in the grammar repo (or against the local `examples/sample.hujson`). Any unknown-node-name error fails the run.
  - Cross-reference each captured node in `queries/highlights.scm` (`string`, `escape_sequence`, `number`, `true`, `false`, `null`, `pair`, `comment`, plus the bracket/punctuation literals) against `src/node-types.json`.
- Confirms H2 if: any query references an absent node.
- Rules out H2 if: all node names resolve and `tree-sitter query` runs cleanly.

**H3 ‚Äî Orphan local `queries/hujson/` and `grammars/hujson/` mislead, but don't cause the symptom.**
- Zed loads queries from the cached external grammar, not from extension-root `queries/hujson/`. The orphan directories are dead code: maintenance hazard, not a runtime cause.
- Evidence: confirmed by file inventory in the diagnostic that produced this addendum (the pinned external grammar's `queries/` is what Zed reads).
- Action regardless of H1/H2 outcomes: delete the orphans (¬ß14.5).

### 14.3 Diagnostic Recipe

Run **before** writing any fix. Captures evidence to assign root cause to H1 vs H2.

```bash
# Set SAMPLE_FILE to the path of your local sample before running this recipe.

SAMPLE_FILE="${SAMPLE_FILE:-../zed-hujson/examples/sample.hujson}"
```

### 14.4 Acceptance Criteria

The fix is complete when **all** of the following hold:

1. **Visual:** opening `examples/sample.hujson` in Zed (any built-in theme) colours each of: `(string)`, `(number)`, `(true)`/`(false)`/`(null)`, `(escape_sequence)`, object keys (the `string` in pair-key position), line comments (`//`), block comments (`/* */`), and the bracket/punctuation tokens `{`, `}`, `[`, `]`, `:`, `,`. The exact palette is theme-driven; the criterion is that each capture *fires*.
2. **Parse:** `tree-sitter parse examples/sample.hujson` against the pinned grammar produces a tree with **zero** `ERROR` nodes.
3. **Query soundness:** `tree-sitter query queries/highlights.scm examples/sample.hujson` runs without "unknown node type" errors at the pinned SHA.
4. **Audit script:** `scripts/check-queries.sh` (this repo) passes locally and in CI against the pinned SHA.
5. **Zed log:** extension load produces no warnings or errors related to grammar fetch, WASM, or query parse.
6. **Repo hygiene:** orphan `grammars/hujson/` and `queries/hujson/` directories are removed from this repo.
7. **Spec hygiene:** ¬ß2.1, ¬ß4.1, ¬ß13 reflect the external-grammar layout (this revision), and the README's developer-workflow section matches.
