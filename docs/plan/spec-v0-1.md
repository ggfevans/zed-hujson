## 1. Overview

### 1.1 Purpose

Build a Zed editor extension that provides native support for **HuJSON** (Human JSON), a superset of JSON defined by the **JWCC** (JSON With Commas and Comments) specification. HuJSON adds exactly two features to standard JSON:

1. **C-style comments** — line comments (`//`) and block comments (`/* */`)
2. **Trailing commas** — optional trailing comma after the last element in arrays and objects

All valid JSON is valid HuJSON. HuJSON intentionally rejects all other extensions (unquoted keys, hex literals, `Infinity`/`NaN`, multi-line strings, etc.).

### 1.2 Scope — v0.1

| In scope | Out of scope (future) |
|---|---|
| Syntax highlighting via Tree-sitter | Formatter integration (`hujsonfmt`) — v0.2 |
| File type association (`.hujson`) | LSP integration — v0.3+ |
| Bracket matching | `.jwcc` file association (reconsider if requested) |
| Auto-indentation | Semantic analysis / validation |
| Code outline (object key navigation) | Tailscale ACL-specific intelligence |

## 2. Architecture

### 2.1 Repository Structure

**Monorepo, grammar at repo root.** The Tree-sitter grammar and the Zed extension live in a single repository (`ggfevans/zed-hujson`), with the grammar files placed at the repo root rather than under a `grammars/hujson/` subfolder. This sidesteps Zed's under-documented `[grammars.*] path` resolution for monorepo grammars (decision 1) and matches the layout of most published Zed grammar extensions.

If the grammar is extracted into its own repo later (e.g. for Neovim/Helix consumers), only `extension.toml` needs to change.

```

**Layout notes:**
- `src/` is intentionally shared between Cargo (`lib.rs`) and tree-sitter (`parser.c`, `grammar.json`, `node-types.json`, `tree_sitter/*`). Cargo only picks up `lib.rs`; the C sources are ignored by `crate-type = ["cdylib"]` and instead consumed by Zed via the grammar bundle.
- `tree-sitter-queries/` (rather than the upstream's `queries/`) holds the standalone grammar queries for Neovim/Helix consumers, so it doesn't shadow Zed's `queries/hujson/`.
- The upstream `bindings/` directory (Rust/Node/Go/Python/Swift) is dropped. We don't need them for the Zed extension; if external consumers later need them they can be regenerated by `tree-sitter generate` on demand.

**`.gitignore` contents:**
```

build/
*.wasm

**`.gitattributes` contents:**
```

src/tree_sitter/* linguist-generated=true
```
This prevents GitHub from showing 100K+ lines of generated C in PR diffs and misreporting the repo as a "C project."

## 3. Tree-sitter Grammar

### 3.1 Approach

**Fork `tree-sitter/tree-sitter-json`** and make a single targeted modification: allow trailing commas in arrays and objects. The upstream grammar already supports `//` and `/* */` comments.

**Fork from HEAD of `master`** (not an older tag). Comment support was added relatively recently — older releases may lack it. Pin the fork to a specific commit SHA in your first commit message for traceability (e.g., "Forked from tree-sitter-json at abc1234"). At time of writing the pinned SHA is `001c28d7a29832b06b0e831ec77845553c89b56d`.

### 3.2 Grammar Configuration Files

After forking, update the following files at the **repo root** (decision 1 — the grammar is not nested under `grammars/hujson/`):

**`tree-sitter.json`** (required for `tree-sitter test` and `tree-sitter parse` to work):
```json
{
  "grammars": [
    {
      "name": "hujson",
      "scope": "source.hujson",
      "file-types": ["hujson"],
      "injection-regex": "hujson",
      "highlights": "queries/highlights.scm"
    }
  ]
}
```

**`package.json`** — inherited from the fork. Update `name` to `tree-sitter-hujson` and `description` accordingly. The rest (dependencies, build scripts) can remain as upstream.

### 3.3 Grammar Change

The modification is to the `commaSep` / `commaSep1` helper functions in `grammar.js` (at the repo root). The exact implementation will depend on the current shape of the upstream helpers at fork time, but the semantic change is:

**Before (standard JSON):**
```javascript
// Only allows: elem, elem, elem
commaSep(rule) {
  return optional(commaSep1(rule));
},
commaSep1(rule) {
  return seq(rule, repeat(seq(',', rule)));
}
```

**After (JWCC — trailing comma tolerated):**
```javascript
// Allows: elem, elem, elem,  (note trailing comma)
commaSep(rule) {
  return optional(seq(rule, repeat(seq(',', rule)), optional(',')));
}
```

### 3.4 Strictness Policy

**Strict HuJSON only.** The grammar MUST NOT accept constructs beyond the JWCC spec:

| Construct | Accept? |
|---|---|
| `//` line comments | ✅ Yes |
| `/* */` block comments | ✅ Yes |
| Trailing commas in `[]` and `{}` | ✅ Yes |
| Standard JSON values (string, number, bool, null) | ✅ Yes |
| Unquoted object keys | ❌ No — parse error |
| Single-quoted strings | ❌ No — parse error |
| Hex number literals (`0xFF`) | ❌ No — parse error |
| `Infinity`, `-Infinity`, `NaN` | ❌ No — parse error |
| Multi-line strings | ❌ No — parse error |
| ES5 identifier property names | ❌ No — parse error |

Since the upstream `tree-sitter-json` already rejects all of these, and our only change is adding trailing comma support, this strictness is inherited automatically — no additional work required.

### 3.5 Error Recovery

Tree-sitter provides built-in error recovery via its GLR parsing algorithm. Invalid tokens are wrapped in `ERROR` nodes in the syntax tree, which Zed renders with error highlighting. No custom error recovery logic is needed for v0.1.

**Expected behaviour on invalid input:**

| Input | Expected result |
|---|---|
| `{key: "value"}` (unquoted key) | `key` wrapped in ERROR node, rest of tree parses |
| `{"a": 0xFF}` (hex literal) | `0xFF` wrapped in ERROR, object structure intact |
| `{"a": 1,, "b": 2}` (double comma) | Second `,` wrapped in ERROR |
| Unterminated `/* comment` | ERROR from `/*` to end of file |
| Nested `/* /* */ */` | First `*/` terminates the comment (C-style, no nesting). The trailing ` */` becomes dangling content → ERROR node |

## 5. Syntax Queries

All query files use Tree-sitter's S-expression format. These should be adapted from Zed's built-in JSONC queries, adjusted for the HuJSON node names produced by the forked grammar.

### 5.1 Highlights (`queries/hujson/highlights.scm`)

```scheme
; Strings
(string) @string
(escape_sequence) @string.escape

; Numbers
(number) @number

; Booleans and null
(true) @constant.builtin
(false) @constant.builtin
(null) @constant.builtin

; Object keys (string node in pair key position)
(pair
  key: (string) @property.json_key)

; Punctuation
"{" @punctuation.bracket
"}" @punctuation.bracket
"[" @punctuation.bracket
"]" @punctuation.bracket
":" @punctuation.delimiter
"," @punctuation.delimiter

; Comments
(comment) @comment
```

> **Note:** The `@property.json_key` capture name is confirmed correct — Zed's built-in JSON and JSONC highlights (`crates/grammars/src/json/highlights.scm`) use this exact name. Node names (`string`, `pair`, `comment`, `escape_sequence`, etc.) should be verified against the actual grammar output after forking and regenerating with `tree-sitter parse` on a sample file.

### 5.2 Brackets (`queries/hujson/brackets.scm`)

```scheme
("{" @open "}" @close)
("[" @open "]" @close)
("\"" @open "\"" @close)
```

### 5.3 Indents (`queries/hujson/indents.scm`)

```scheme
(object) @indent
(array) @indent

"}" @outdent
"]" @outdent
```

### 5.4 Outline (`queries/hujson/outline.scm`)

```scheme
(pair
  key: (string) @name) @item
```

This provides a navigable outline of top-level and nested object keys in Zed's outline panel.

## 6. Data Handling

### 6.1 Input

The extension processes text buffers in the Zed editor. The Tree-sitter parser operates on the raw UTF-8 byte stream. No file I/O is performed by the extension itself — Zed handles file reading/writing.

### 6.2 Character Encoding

Tree-sitter and Zed both operate on UTF-8. HuJSON (like JSON per RFC 8259) requires UTF-8 encoding. No special encoding handling is needed.

**Limitation:** If a user opens a UTF-16 or UTF-32 encoded `.hujson` file (rare but possible with legacy tooling), Tree-sitter will silently misparse it — producing garbled output with no useful error. This is a Zed-wide limitation, not specific to this extension. No mitigation needed in v0.1, but worth noting for awareness.

### 6.3 Large Files

Tree-sitter's incremental parsing handles large files efficiently — only changed regions are re-parsed. No file size limits need to be imposed by the extension. Zed's own large-file handling applies.

## 7. Error Handling Strategy

### 7.1 Grammar-Level Errors

Handled entirely by Tree-sitter's error recovery (see §3.5). Invalid tokens produce `ERROR` nodes that Zed highlights in red. No custom logic required.

### 7.2 Extension-Level Errors

The v0.1 extension has no runtime logic beyond registration — there are no extension-level errors to handle. When formatter/LSP support is added in v0.2+, error handling for process spawning, timeouts, and malformed output will be needed.

### 7.3 Grammar Namespace Collision

The extension registers a grammar named `hujson`, distinct from Zed's built-in `json` and `jsonc` grammars. Since the file extension (`.hujson`) is unique, there should be no conflict — Zed dispatches grammars by file extension, not by grammar family.

**Decision 5 (resolved): defence-in-depth verification.** Confirm via three layers, in order:

1. **Manual smoke in Zed** — open `.json`, `.jsonc`, and `.hujson` files; confirm each binds to the correct grammar via the status bar; confirm Zed's log shows no duplicate-grammar warnings.
2. **README "Compatibility" section** — explicitly state that this extension only handles `.hujson`, that `.json`/`.jsonc` are left to Zed's built-ins, and that the grammar id `hujson` is distinct from `json`/`jsonc`. This sets expectations for users and registry reviewers.
3. **Registry pre-publish search** — before opening the PR to `zed-industries/extensions`, grep the registry for `hujson` (id collision), `\.hujson\b`, and `path_suffixes.*hujson` (extension claim). If any prior claim exists, decide on rename / coordination before submitting.

### 7.4 Grammar Generation Errors

If `tree-sitter generate` fails after modifying `grammar.js`, the grammar change has introduced a conflict. Debug with:

```bash
tree-sitter generate --log
tree-sitter parse test-file.hujson --debug
```

CI should catch these before merge (see §9).

## 8. Testing Plan

### 8.1 Grammar Tests (Tree-sitter Corpus)

Tree-sitter uses a corpus-based testing format. Each test file in `test/corpus/` (at the repo root, per decision 1 — not under `grammars/hujson/test/corpus/`) contains named test cases with input and expected syntax tree output.

**Decision 4 (resolved): node-name verification belt-and-braces.** After every `tree-sitter generate`:

1. Audit `src/node-types.json` against every node referenced in `queries/hujson/*.scm` (e.g. `string`, `pair`, `comment`, `escape_sequence`, `number`, `true`, `false`, `null`, `object`, `array`).
2. Run `tree-sitter query queries/hujson/<file>.scm` for each query file; any unknown node name fails the build.

Both checks run locally (via `scripts/check-queries.sh`) and in CI (§9.1).

#### Required Test Cases

**`comments.txt`** — Comment handling:
- Line comment at end of line
- Line comment on its own line
- Block comment inline
- Block comment spanning multiple lines
- Comment inside an object (between pairs)
- Comment inside an array (between elements)
- Comment before/after root value
- Nested `/*` inside block comment (C-style, no nesting — first `*/` terminates, remainder is ERROR)
- Empty block comment `/* */`

**`trailing_commas.txt`** — The core HuJSON addition:
- Trailing comma in object `{"a": 1,}`
- Trailing comma in array `[1, 2, 3,]`
- Trailing comma in nested object `{"a": {"b": 1,},}`
- Trailing comma in nested array `[[1,], [2,],]`
- Trailing comma with comment after `{"a": 1, // note}`
- Comment before trailing comma (comma on next line after comment — tests `extras`/whitespace interaction):
  ```json
  {
    "a": 1 // comment
    ,
  }
  ```
- Trailing comma in empty-ish structures — `{"a": 1,}` (single element + trailing)
- Multiple trailing commas `{"a": 1,,}` → ERROR (only one trailing comma allowed)
- Leading comma `{,"a": 1}` → ERROR
- Trailing comma in empty object `{,}` → ERROR (no element to trail after)
- Trailing comma in empty array `[,]` → ERROR (no element to trail after)

**`literals.txt`** — Standard JSON value parsing (inherited from upstream, verify not broken):
- Strings with escape sequences (`\"`, `\\`, `\n`, `\uXXXX`)
- Numbers (integer, float, negative, exponent)
- `true`, `false`, `null`

**`objects.txt`** — Object structures:
- Empty object `{}`
- Single pair
- Multiple pairs
- Nested objects
- Object with all value types

**`arrays.txt`** — Array structures:
- Empty array `[]`
- Single element
- Multiple elements
- Nested arrays
- Mixed value types

**`degenerate.txt`** — Edge cases and boundary conditions:
- Empty file (0 bytes) — should parse without crash, produce empty tree
- Whitespace-only file — should parse cleanly, no ERROR nodes
- Comment-only file (`// just a comment`, no JSON value) — valid HuJSON per spec? Verify against `hujsonfmt` behaviour
- BOM (Byte Order Mark `\uFEFF`) at start of file — Tree-sitter should handle or ERROR gracefully
- Bare root value: `"hello"` (string, no object/array wrapper) — valid JSON, valid HuJSON
- Bare root value: `42` (number, no wrapper) — valid JSON, valid HuJSON
- Bare root value: `null` — valid JSON, valid HuJSON
- Unicode in comments: `// 日本語コメント 🎉`
- Unicode in string values: `{"emoji": "🎯", "cjk": "漢字"}`
- Deeply nested structure (50+ levels) — verify parser doesn't stack overflow
- Very long single line (>100K characters) — verify incremental parsing handles it

**`invalid.txt`** — Strictness enforcement (all should produce ERROR nodes):
- Unquoted key: `{key: "value"}`
- Single-quoted string: `{'key': 'value'}`
- Hex literal: `{"a": 0xFF}`
- Infinity: `{"a": Infinity}`
- NaN: `{"a": NaN}`
- Unquoted string value: `{"a": hello}`

### 8.2 Integration Tests

Manual testing in Zed with real-world files:

1. **Tailscale ACL policy** — a real `.hujson` file with comments and trailing commas
2. **Large file** — a `.hujson` file >10,000 lines (generate synthetically) to verify incremental parsing performance
3. **Mixed valid/invalid** — a file with intentional errors to verify error recovery doesn't break the rest of the tree
4. **File association** — verify that opening a `.hujson` file automatically activates HuJSON highlighting

### 8.3 Visual Regression

After initial development, capture screenshots of highlighting for the key test files. These serve as a reference for future changes. Not automated in v0.1 — just eyeball it.

### 8.4 Comparison Baseline

Open a `.hujson` file in Zed, then click the language name in the status bar (bottom right) and select "JSON with Comments" to force JSONC mode. Compare against the default HuJSON highlighting. Verify:
- Comments highlight identically
- Trailing commas are NOT errors in HuJSON mode (but ARE errors in JSONC mode)
- All other highlighting matches

### 11.3 Upstream Sync Strategy

The grammar is a fork of `tree-sitter/tree-sitter-json`. Upstream changes (bug fixes, performance improvements, Tree-sitter API updates) need to be merged periodically.

**Process:**

Since the grammar lives in a subfolder of a monorepo, you cannot directly `git merge` the upstream repo — the directory structures don't align. Instead:

1. Clone upstream separately for comparison: `git clone https://github.com/tree-sitter/tree-sitter-json.git /tmp/tree-sitter-json`
2. Diff the upstream changes against your fork: `diff -ru /tmp/tree-sitter-json/grammar.js grammars/hujson/grammar.js`
3. Manually port relevant changes into `grammars/hujson/`, preserving the trailing comma modification
4. Re-run `tree-sitter generate` and the full test corpus after any port
5. Commit with a message referencing the upstream commit: "Sync grammar with tree-sitter-json at abc1234"

**Alternative:** If sync frequency increases, consider using `git subtree` to manage the grammar subfolder, which allows direct merging: `git subtree pull --prefix=grammars/hujson upstream-json master --squash`. This requires initial setup with `git subtree add` but simplifies future syncs.

**What could break on upstream sync:**
- **Node name changes** — would break all `.scm` query files. Grep queries for renamed nodes.
- **`extras` array changes** — could affect comment handling. Verify comments still parse.
- **`commaSep` refactors** — our core change. If upstream restructures these helpers, manually reapply the trailing comma logic.
- **`tree-sitter.json` changes** — this config file controls scope, file-types, and highlights path. Changes could affect parser generation or Neovim/Helix consumers.
- **Tree-sitter CLI version bumps** — may require regenerating `src/parser.c` with a newer CLI.

**Cadence:** Check upstream quarterly, or when a Tree-sitter major version is released. Set a GitHub reminder or Dependabot-style alert.

### 11.4 Grammar Extraction

If other editors request the Tree-sitter grammar:
1. Extract `grammars/hujson/` into `ggfevans/tree-sitter-hujson`
2. Update `extension.toml` grammar reference to point to the new repo
3. Publish to npm as `tree-sitter-hujson` for Neovim/Helix consumption

# Clone

### 12.3 Development Loop

1. Edit `grammars/hujson/grammar.js`
2. Run `tree-sitter generate` (in `grammars/hujson/`)
3. Run `tree-sitter test` to verify corpus
4. Run `tree-sitter parse test-file.hujson` to inspect tree
5. Rebuild WASM: `cargo build --release --target wasm32-wasip2`
6. Reload extension: open command palette (`Cmd/Ctrl+Shift+P`) → `zed: reload extensions` (faster than restarting Zed)
7. Open a `.hujson` file and verify highlighting

### 13.0 Open-question resolutions (2026-04-09)

Five items the spec previously flagged as "verify at implementation time" have been resolved:

| # | Topic | Decision |
|---|---|---|
| 1 | Grammar location | **Repo root** (no `grammars/hujson/` subfolder). `extension.toml` omits the `path` key. Avoids unverified monorepo `path` behaviour. |
| 2 | `zed_extension_api` version | Pin to **latest stable** at implementation time (check crates.io); update spec §4.4 with the value actually used. |
| 3 | `Extension` trait shape | Add **minimal no-op stubs** for any required methods beyond `new()`, returning the most conservative unsupported value. v0.1 stays highlighting-only. |
| 4 | Node-name verification | **Belt-and-braces** — audit `src/node-types.json` AND run `tree-sitter query` per `.scm` file, both locally (`scripts/check-queries.sh`) and in CI. |
| 5 | `.hujson` file conflict | **Three layers** — manual smoke in Zed, README "Compatibility" section, and a registry-wide search before publishing. |

### 13.1 Original log

| Decision | Choice | Rationale |
|---|---|---|
| Repo structure | Monorepo (grammar + extension), split later if needed | Minimises overhead for solo development; grammar subfolder is self-contained and extractable |
| Grammar approach | Fork `tree-sitter-json` | Upstream already has comment support; only trailing comma change needed (~5 lines) |
| Strictness | Strict HuJSON only | Matches the JWCC spec; rejects JSON5 extras as parse errors. Inherited from upstream grammar — no extra work |
| File extensions | `.hujson` only (v0.1) | Primary extension in practice; `.jwcc` can be added trivially if requested |
| Licence | MIT | Matches upstream `tree-sitter-json`; maximally permissive |
| GitHub owner | `ggfevans` (personal) | Fast to ship; can transfer to org later |
| Formatter | Out of scope for v0.1 | Scope creep guard — ship highlighting first |
| LSP | Out of scope (v0.3+, maybe never) | Significant effort; unclear value for a config format |
| Indentation | 2-space, soft tabs | JSON/HuJSON convention; set explicitly in config.toml |
| Word characters | Include `-` (hyphen) | Double-click selects full hyphenated keys like `acl-rule-name`. Matches config file UX expectations |
| Edge case testing | Comprehensive (degenerate files, Unicode, BOM) | Catches real-world weirdness; low effort to add to corpus |
